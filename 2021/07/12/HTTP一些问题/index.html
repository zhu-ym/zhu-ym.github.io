    HTTP一些问题

    发表于 2021-07-12 更新于 2021-07-17 分类于 计算机网络

    发现对HTTP的整体架构了解较少，总结了一些问题

    HTTP报文信息

    HTTP请求报文

    HTTP请求报文主要由请求行+请求头部+请求数据组成

    请求行主要是包含了请求方法、URL、和HTTP协议版本的信息

    这里就延伸出了关于请求方法的一个问题：

    Q1:请求行常见方法？get和post方法对比

    HTTP的请求方法有get获取资源、post传输数据、head获取报文首部、options查看服务器支持方法、put上传、delete删除等方法

    其中最常用的就是get和post

    get常用来请求资源，其参数连接在URL之后，经指定的资源经服务器端解析后返回响应内容

    而post但更重要的目的是为了传输一些主体参数，比如我们一般提交表单信息时，常用的是post方法

    两者主要区别：

    • 首先是语义上的区别，一个是请求资源，类似从服务器上拉资源过来，一个是传递数据，提交表单，主要在于将数据推到服务器那边

    • 语义上的差别也带来了幂等性的问题，get方法一般对服务器读请求，无副作用，多次get操作一般上幂等的，而post方法很多时候在服务器上是写请求，重复提交表单可能会在数据库插入多条数据，这一般要求服务器的接口进行验证实现幂等性

    • get方法主要是读取、再加上其幂等性导致可以对响应的数据做缓存，post不幂等也就意味着不能随意多次执行。因此一般也就不会主动响应数据。

    • 最后的话因为get方法参数会拼接做url上，太长的url会影响具体服务器对其进行解析，所以url长度一般有限制，从而get参数也有限制

    了解请求行后，我们就到了请求头部，请求头部包含一些字段和对应值，描述了一些报文的信息。比如Content-length表明了实体的大小，Connection 控制着长短连接 ，user agent包含了HTTP 客户端程序的信息，以及缓存控制、编码控制等一系列的信息

    请求行结束之后，最后就是实体内容，从上我们可以发现，请求头部到请求数据之间空了一行，这个地方又引出一个问题：

    Q2:TCP是面向字节流的协议，那http报文经过TCP传输后，怎么防止粘包

    我们知道TCP的Nagle算法，会让TCP的组包时，避免发送小的报文段，会将收集到的小数据整合再发送，这可能会导致两个HTTP紧紧连接在一起，那解析这样的TCP报文时，如何区分两个HTTP

    我认为两个关键点，一个是Content-type，指明了实体的大小，另外一点是特殊标记符号，换行符。

    换行符可以让我们清楚哪里是实体开始的地方，实体大小让我们清楚实体什么时候结束，实体结束后面就是另一个HTTP报文的开始，通过这些数据，我们知道HTTP报文开始结束，能正确解析出来，也就不会粘包了

    HTTP响应报文

    与HTTP请求报文对应，HTTP也有响应报文，响应报文一般包含：响应行+响应首部+响应数据

    响应行包含了：包含了状态码以及原因短语，用来告知客户端请求的结果。

    这里需要需要我们了解常见状态码

    Q3:常见状态码

    下面是一些常用状态码

    2XX:成功

    200 OK 正常处理

    204 No Content 请求处理成功，但是没有资源可返回

    206 Partial Content 客户端范围请求，请求成功

    3XX：重定向

    301 永久重定向 资源分配了新的URI，以后使用该URI

    302 临时重定向 希望用户本次访问使用该URI

    303 See Other 功能和302类似，但制定客户端用GET方法访问

    304 Not Modified 如果请求报文首部包含一些条件，例如：If-Match，If-Modified-Since等如果不满足条件，则服务器会返回 304 状态码。

    307 临时重定向，与 302 的含义类似，但是 307 要求浏览器不会把重定向请求的 POST 方法改成 GET 方法。

    4XX：客户端错误

    400 请求报文语法错误

    401 HTTP认证

    403 服务器拒绝访问

    404 没有请求资源

    5XX：服务器错误

    500 服务器内部执行请求发生错误

    502 网关错误

    503 服务器超负载或者停机

    对于HTTP响应报文的其他信息，和请求报文差不多

    至此我们对HTTP报文有了了解，那么关于HTTP本身，它到底是怎么，有哪些内容呢

    HTTP特性

    HTTP协议用于客户端和服务端之间，通过请求和响应报文的交换达成通信，是一种无状态的协议。

    针对这个无状态，也有一个问题：

    Q4:如果想要HTTP保存状态该怎么办

    HTTP1.1引入Cookie来保存状态信息

    Cookie 是服务器发送到用户浏览器并保存在本地的一小块数据，它会在浏览器之后向同一服务器再次发起请求时被携带上，用于告知服务端两个请求是否来自同一浏览器。

    服务器发送的响应报文包含 Set-Cookie 首部字段，客户端得到响应报文后把 Cookie 内容保存到浏览器中。

    客户端之后对同一个服务器发送请求时，会从浏览器中取出 Cookie 信息并通过 Cookie 请求首部字段发送给服务器。

    一般注意cookie会存在持久性，作用域上的问题

    当Cookie被浏览器禁用时，可以就需要将信息存储到服务端，我们称为Session，再而是使用 URL 重写技术，将 Session ID 作为 浏览器URL 的参数进行传递，到服务器上根据SessionID 查找具体到状态信息

    回到HTTP的其他特性，HTTP的特性其实和HTTP版本息息相关，这里可以延伸出多个问题：

    Q5:HTTP 1.0和1.1版本的区别

    主要有几点：

    • 1.0使用短连接，每次发送数据都有TCP的连接和挥手，效率比较低，1.1后默认使用长连接，只需一次建立就可以传输多次数据，传输完成后，只需要一次切断连接即可。长连接的连接时长可以通过请求头中的 keep-alive 来设置
    • 1.0认为每个机器只能绑定一个ip，所以那时的url没有传递主机名，1.1后使用了虚拟网络，一台物理机上可以存在多个虚拟主机，他们共享一个ip地址
    • 1.0仅提供了最基本的认证，很容易收到窥探，1.1后使用了摘要算法来进行身份验证
    • 1.0 不支持断点续传，1.1 后支持断点续传，通过使用请求头中的 Range 来实现。
    • 1.1引入cookie来保存状态

    Q6:HTTP1.1和2.0的区别

    前面已经了解1.1，这里主要介绍2.0的新改变：

    • 强化安全：http2.0一般跑在https上
    • 二进制格式：HTTP2.0使用了更接近TCP/IP的二进制格式抛弃了ASCII码，提高了解析效率
    • 多路复用：即每一个请求都是是用作连接共享。一个请求对应一个id，这样一个连接上可以有多个请求。
    • 头部压缩：由于 HTTP 1.1 经常会出现 User-Agent、Cookie、Accept、Server、Range 等字段可能会占用几百甚至几千字节，导致头部偏重，HTTP 2.0 使用 HPACK 算法进行压缩

    总的来说，http应用广泛，易于拓展，但是除了2.0外，在安全性方面还是存在很多问题：

    • 数据明文传输，可能会被窃听
    • 通信身份可能会被伪造
    • 无法证明报文完整性，报文完整性可能被篡改

    而这些，可以通过HTTPS协议来解决

    HTTPS

    前面提到2.o为了强化安全，一般跑在https上，那么https的原理是什么，为什么能保证安全？

    HTTPS实际上是HTTP + SSL(secure socket layer)

    之前是HTTP直接和TCP通信，在HTTPS里面，HTTP先和SSL通信，再由SSL和TCP通信，通过使用 SSL，HTTPS 具有了加密（防窃听）、认证（防伪装）和完整性保护（防篡改）的功能，安全性大大提高。

    这里针对这几个方案的解决原理，也有几个问题：

    Q7:HTTPS加密手段

    我们常见的加密手段有：

    • 对称加密：通信双方共用一个密钥对传输的数据进行加密，解锁。但是在传输这个密钥的时候，显然还是可能被窃取到
    • 非对称加密：服务器有两个密钥，一个是公钥，一个是私钥，公钥和私钥可以互解对方加密的数据。采用这个加密时，服务器将自己的公钥明文发送给客户端，客户端用公钥发送的消息只有服务端的私钥才能解密，所以这一面通信是安全的，但是第一步别人可以窃取公钥，服务端到客户端的信息就不是安全的

    这里需要注意一点，对称加密算法简单，非对称加密算法比较耗时

    到这，两种算法好像都不可行，但是我们发现了一点，非对称加密能解决单向的通信安全

    很容易得出我们可以搞两个方向的非对称加密实现双向安全通信，可以但是没有必要，HTTPS采用了对称加密和非对称加密的混合加密方式，运算速度比前者方更快

    混合加密方式：

    首先采用非对称加密的方式，保证客户端到服务端的安全通信，然后客户端通信事，将自己生成的密钥传递给服务端，最后两者都用这个密钥对称加密通信。

    前面貌似混合加密方式已经很完美，但是这个方案和双非对称方案都还存在一个问题：中间人攻击

    • 某网站有用于非对称加密的公钥A、私钥A’。

    • 浏览器向网站服务器请求，服务器把公钥A明文给传输浏览器。

    • 中间人劫持到公钥A，保存下来，把数据包中的公钥A替换成自己伪造的公钥B（它当然也拥有公钥B对应的私钥B’）。

    • 浏览器生成一个用于对称加密的密钥X，用公钥B（浏览器无法得知公钥被替换了）加密后传给服务器。

    • 中间人劫持后用私钥B’解密得到密钥X，再用公钥A加密后传给服务器。

    • 服务器拿到后用私钥A’解密得到密钥X。

    之后两者再通信，却有一个中间人拿到他们的密钥X，通信业不太安全

    这个问题的本质在于，中间人自制的公钥B浏览器判断不出来，浏览器无法确认收到的公钥是不是网站自己的

    这个问题的解决设计到HTTPS的另一个特性：认证

    Q8：认证

    通过使用 证书 来对通信方进行认证。认证可以保证浏览器判断公钥是不是网站自己的！

    数字证书认证机构（CA，Certificate Authority）是客户端与服务器双方都可信赖的第三方机构。

    服务器的运营人员向 CA 提出公开密钥的申请，CA 在判明提出申请者的身份之后，会对已申请的公开密钥做数字签名，然后分配这个已签名的公开密钥，并将该公开密钥放入公开密钥证书后绑定在一起。

    进行 HTTPS 通信时，服务器会把证书发送给客户端。客户端取得其中的公开密钥之后，先使用数字签名进行验证，如果验证通过，就可以开始通信了。

    至此，加密的安全性也才能完全保证

    最后SSL 提供报文摘要功能来进行完整性保护。

    HTTP 也提供了 MD5 报文摘要功能，但不是安全的。例如报文内容被篡改之后，同时重新计算 MD5 的值，通信接收方是无法意识到发生了篡改。

    HTTPS 的报文摘要功能之所以安全，是因为它结合了加密和认证这两个操作。试想一下，加密之后的报文，遭到篡改之后，也很难重新计算报文摘要，因为无法轻易获取明文。
